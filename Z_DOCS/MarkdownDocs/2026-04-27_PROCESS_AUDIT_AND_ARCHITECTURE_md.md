Taking this one step at a time, here are the essential elements to retain, the dead ends to discard, and the final code base.
1. The Objectives & Definitions (The "Why")
This section locks in the vocabulary and prevents an AI from reverting to standard public relations terminology.
* The Objective (The Product): To evaluate public education as a manufacturing process. The mandated product is a "Standard-Compliant Unit" (a young adult capable of career, college, and civic engagement).
* The Measurement Proxy (The Bottleneck): A standard-compliant unit must survive quality assurance (QA) checks in ELA, Science, and Math. Because Math consistently has the lowest pass rate, it acts as the mathematical ceiling. We use the 10th-grade Math SBAC as the absolute "Best-Case Scenario" proxy for the system's yield.
* Excommunicated Verbiage: Do not use "College Ready" (the SBAC is a shortened 24-question algebraic threshold, not a comprehensive college prep exam). Do not use "Cost of a Math-Proficient Student" (Math is just the limiting proxy, not the sole objective).
* The Financial Input (Closing the Trap Door): To calculate the Cost of Poor Quality (COPQ), the primary financial metric isolates Program 01 (General Education) Certificated Salaries. This preemptively shuts down the bureaucratic defense that high costs are due to Special Education, transportation, or facility overhead.
2. The Data Architecture (The "How")
This section defines the pipeline and how the "Smash" (the join between finance and assessment) operates.
* The Primary Key: The linking mechanism between all datasets is the 5-digit CODIST (County-District Code, e.g., 34002 for Yelm).
* The Master Index: The AI does not read hundreds of CSVs. It reads the EdData_Master_Index.csv, which acts as the "Card Catalog," containing the EntityKey, Organization Level (State, District, School), and the exact file paths for both the Financial and Assessment records.
* Data Storage: * S275_APP_READY: Contains the standardized financial pipeline files. Names/PII are strictly redacted.
o ASSMT_APP_READY: Contains the parsed assessment yield files.
3. The "Wrong Paths" (What Was Rejected & Why)
Documenting failed logic is critical so an AI doesn't try to "optimize" the process by reverting to a bad idea.
* Rejected Path: The 3-Stage Financial Cleaning (Staging Folders). * What we tried: Extracting raw data to a staging folder, running a header audit, and then moving to a finalized folder.
o Why we killed it: It created digital "Muda" (waste), consumed massive disk space, and created three versions of the truth.
o The Fix: The "One-Step Master Extractor" connects directly to the raw Access DBs, standardizes the 51/52/54 column variations on the fly, redacts names, and outputs directly to the APP_READY folder.
* Rejected Path: Filtering Assessment Data to "10th Grade Only".
o What we tried: Stripping out grades 3-8 to focus solely on the final 10th-grade QA check.
o Why we killed it: It destroyed the ability to calculate "Yield Decay" (tracking a cohort as it moves from elementary to high school to see where the system loses control of the product).
o The Fix: The Assessment Parser flattens all grade levels but retains the specific L1 through L4 defect counts to allow deep root-cause analysis on defect severity.
* Rejected Path: Relying on the OSPI OrganizationID.
o What we tried: Using the internal random index generated by OSPI's modern dashboard.
o Why we killed it: The internal index does not match the legacy S-275 CODIST.
o The Fix: Restored the DistrictCode and SchoolCode to the SQL extraction stored procedure to ensure a perfect 1:1 join without needing a static crosswalk table.
4. The Final Code Base
These are the two engine scripts that execute the logic above. Storing these in the governance folder ensures the pipeline can be rebuilt at any time.
Script 1: The S-275 One-Step Extractor
* Filename: Extract_S275_Master.ps1
* Purpose: Connects to .accdb files, filters for prog = '01' and certfte > 0, redacts PII, forces standardization to the 5-digit CODIST, and populates S275_APP_READY.
Script 2: The Multi-Level Assessment Parser & Indexer
* Filename: Build-EdData-Smash.ps1
* Purpose: Reads the raw GEMAI__AssmtGroup files. Executes the "Routing Switch" to splinter data into State (00000), District (34002), and School (34002-4158) levels. Calculates the hard counts of L1 (Severe Defect) through L4. Populates ASSMT_APP_READY and builds EdData_Master_Index.csv.
